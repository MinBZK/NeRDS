# 3. Work transparently and use open source

Open source is a way of working where the creators make the source code, documentation and design publicly available under an open source license. This allows anyone to view, copy, modify or share the software. People can make suggestions for improvements or additions. Sharing code and using open source software can lead to better, safer and more cost-effective solutions.

## Why is it important?

- **Transparency**: Iedereen kan zien hoe de software werkt en hoe besluiten worden genomen
- **Cost effectiveness**: Vermijd licentiekosten en afhankelijkheid van één leverancier
- **Cooperation**: Organisaties kunnen kennis delen en van elkaar learn
- **Flexibility**: Aanpassingen zijn mogelijk zonder afhankelijk te zijn van externe leveranciers
- **Security**: Problemen kunnen sneller worden ontdekt en opgelost dankzij gemeenschappelijke inspectie
- **Sustainability**: Voorkomt vendor lock-in en maakt langetermijnbeheer mogelijk
- **Innovation**: Lost gemeenschappelijke problemen op met beschikbare technologie
- **Time and cost saving**: Maakt gebruik van bestaande oplossingen in plaats van het wiel opnieuw uit te vinden

By working open and transparent and embracing open source software, you contribute to a more transparent, flexible and responsible digital government.

### Implementation steps

1. **Evaluate existing solutions:** Onderzoek eerst of er al geschikte open source alternatieven bestaan.
2. **Publish Your Own Code**: Maak de broncode van je software openbaar, tenzij er specifieke redenen zijn om dit niet te doen
3. **Use Open Standards**: Zorg ervoor dat je systemen kunnen communiceren met andere systemen (zie [Gebruik open standaarden](../open-standaarden/index.md))
4. **Document throughly**: Maak het voor anderen gemakkelijk om je code te begrijpen en eraan bij te dragen
5. **Building a community**: Moedig contributing en feedback aan op je projecten (zie [Deel, hergebruik en werk samen](../samenwerking/index.md))

### Benefits of publishing your own code

- **Better documentation**: Stimuleert duidelijkere documentatie
- **Cleaner code**: Leidt tot nettere, beter onderhoudbare code
- **Data security**: Zorgt voor duidelijkheid rond beschermde gegevens
- **Improvement**: Creëert mogelijkheden voor suggesties ter verbetering

### Considerations when choosing open source technology

- **Does it measure user needs?** Beantwoordt de oplossing aan de specifieke eisen?
- **What are the total costs?** Inclusief implementatie, migratie en ondersteuning
- **How mature is the code?** Is het een stabiel en actief onderhouden project?
- **What level of support exists?** Is er een actieve gemeenschap of commerciële ondersteuning?
- **Is it secure enough?** Controleer veiligheidsgeschiedenis en updatefrequentie
- **Is the license acceptable?** Begrijp de licentievoorwaarden en -verplichtingen

### When you open your own code

- Make sure no sensitive information is published (such as passwords and API keys)
- Ensure clear README with installation and contribution instructions
- Choose a suitable open source license (see below)
- Keep the documentation active
- Be responsive to issues and pull requests

### Recommended license for public projects

Voor Dutch overheidsprojecten wordt de **European Union Public Licence (EUPL v1.2)** aanbevolen. Deze licentie:

- Has been specifically developed for European public software
- Is legally validated in all EU languages
- Has a "weak reciprocal" character (changes in original code must be shared under the same license)
- Compatible with other popular open source licenses
- Meets the open source definition

Bij het publiceren van overheidscode is het raadzaam de EUPL-1.2 te gebruiken, tenzij er specifieke redenen zijn om een andere licentie te kiezen. Raadpleeg de [OSPO-NL kennisbank over OSS-licenties](https://ospo-nl.github.io/kennisbank/best-practices/open-source-software-licenties/) voor meer informatie over licentietypes en overwegingen.

## Related Tools

- [Opensourcewerken community](https://developer.overheid.nl/communities/open-source-werken) - Community van het Ministry van Binnenlandse Zaken voor praktische kennis, een forum om ervaringen te delen en informatie over relevante regelgeving voor open source werken.

- [OSPO-NL Kennisbank](https://ospo-nl.github.io/kennisbank/) - Verzameling van kennis over het opzetten van een Open Source Program Office (OSPO) binnen de Dutch government, met praktische handleidingen en guidelines voor het werken met open source.

- [Instructie Rijksdienst bij aanschaf ICT-diensten of ICT-producten](https://wetten.overheid.nl/BWBR0024717/2008-11-23) - Officiële instructie die voorschrijft dat bij de aanschaf van ICT-diensten of -producten door de Central Government, open standaarden moeten worden geëist.
